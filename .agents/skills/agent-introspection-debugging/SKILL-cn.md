# Agent内省调试

当agent运行反复失败、无进展消耗token、循环调用相同工具，或偏离预期任务时使用此技能。

这是一项workflow技能，而非隐藏运行时。它指导agent在上报给人类之前系统性地自行调试。

## 何时激活

- 达到工具调用/循环上限故障
- 多次重试但无任何进展
- 上下文膨胀或prompt漂移导致输出质量开始下降
- 文件系统或环境状态与预期不符
- 通过诊断和小型纠正操作大概率可恢复的工具故障

## 适用范围

适用场景：
- 在盲目重试前捕获故障状态
- 诊断常见的agent专属故障模式
- 执行受控恢复操作
- 生成结构化、人类可读的调试报告

不可用于以下场景的主流程：
- 代码变更后的功能验证，请使用`verification-loop`
- 已有更细分的ECC技能可用于特定框架调试时
- 当前运行框架无法自动执行的运行时承诺

## 四阶段循环

### 阶段1：故障捕获

在尝试恢复前，精准记录故障信息。

需捕获的信息：
- 错误类型、消息，以及可用时的堆栈跟踪
- 最近一次有效的工具调用序列
- agent当时正在执行的操作
- 当前上下文压力：重复的prompt、过大的粘贴日志、重复的计划或失控的注释
- 当前环境假设：cwd、分支、相关服务状态、预期存在的文件

最小捕获模板：

```markdown
## 故障捕获
- 会话/任务：
- 进行中的目标：
- 错误：
- 上一个成功步骤：
- 上一个失败的工具/命令：
- 观察到的重复模式：
- 需要验证的环境假设：
```

### 阶段2：根因诊断

在进行任何变更前，将故障与已知模式匹配。

| 模式 | 可能原因 | 检查项 |
| --- | --- | --- |
| 工具调用次数达上限/重复执行相同命令 | 循环或无退出的观察路径 | 检查最近N次工具调用是否存在重复 |
| 上下文溢出/推理能力下降 | 无限制的注释、重复的计划、过大的日志 | 检查最近的上下文是否存在重复和低信息量内容 |
| `ECONNREFUSED` / 超时 | 服务不可用或端口错误 | 验证服务健康状态、URL和端口假设是否正确 |
| `429` / 配额耗尽 | 重试风暴或缺少退避机制 | 统计重复调用次数，检查重试间隔 |
| 写入后文件缺失/过时的diff | 竞态、cwd错误或分支漂移 | 重新检查路径、cwd、git状态和文件实际是否存在 |
| “修复”后测试仍失败 | 假设错误 | 隔离出确切失败的测试，重新推导bug原因 |

诊断问题：
- 这是逻辑故障、状态故障、环境故障还是策略故障？
- agent是否丢失了真实目标，开始优化错误的子任务？
- 故障是确定性的还是偶发的？
- 可验证诊断结果的最小可逆操作是什么？

### 阶段3：受控恢复

使用可改变诊断结果的最小操作进行恢复。

安全恢复操作：
- 停止重复重试，重新陈述假设
- 裁剪低信息量上下文，仅保留当前目标、阻塞点和证据
- 重新检查实际的文件系统/分支/进程状态
- 将任务范围缩小到单个失败的命令、单个文件或单个测试
- 从推测性推理切换到直接观察
- 当故障风险高或被外部因素阻塞时，上报给人类

不要声称执行了“重置agent状态”或“更新运行框架配置”等不支持的自动修复操作，除非你确实通过当前环境中的真实工具执行了这些操作。

受控恢复检查清单：

```markdown
## 恢复操作
- 选定的诊断结果：
- 执行的最小操作：
- 操作安全性说明：
- 可证明修复生效的证据：
```

### 阶段4：内省报告

最后生成报告，让后续agent或人类可以清晰了解恢复过程。

```markdown
## Agent自调试报告
- 会话/任务：
- 故障：
- 根因：
- 恢复操作：
- 结果：成功 | 部分成功 | 阻塞
- Token / 时间消耗风险：
- 需要的后续操作：
- 后续需固化的预防性变更：
```

## 恢复启发式规则

优先按以下顺序执行干预措施：

1. 用一句话重述真实目标。
2. 验证实际状态，不要信任记忆。
3. 缩小故障范围。
4. 执行一次可区分根因的检查。
5. 仅在完成以上步骤后重试。

错误模式：
- 对相同操作稍加改述后重试三次

正确模式：
- 捕获故障
- 分类模式
- 执行一次直接检查
- 仅当检查结果支持时才变更计划

## 与ECC的集成

- 如果代码发生了变更，恢复后使用`verification-loop`。
- 当故障模式值得转化为直觉或后续技能时，使用`continuous-learning-v2`。
- 当问题不是技术故障而是决策模糊时，使用`council`。
- 如果故障来自本地状态冲突或仓库漂移，使用`workspace-surface-audit`。

## 输出标准

启用此技能时，不要仅以“我修复了问题”作为结束。

始终需要提供：
- 故障模式
- 根因假设
- 恢复操作
- 证明情况已改善或仍处于阻塞状态的证据

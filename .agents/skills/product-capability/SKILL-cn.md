# 产品能力

本技能将产品意图转化为明确的工程约束。

当问题不再是「我们应该构建什么？」而是「在开始实施前必须明确哪些确切的条件？」时使用。

## 适用场景

- 已有PRD、路线图项、讨论或创始人笔记，但实施约束仍然不明确
- 功能涉及多个服务、代码仓库或团队，需要在编码前确定能力契约
- 产品意图清晰，但架构、数据、生命周期或政策影响仍然模糊
- 高级工程师在评审过程中不断重复相同的隐含假设
- 你需要一个可在不同工具和会话中复用的持久化产物

## 标准产物

如果代码仓库有持久化的产品上下文文件，例如`PRODUCT.md`、`docs/product/`或项目规范目录，在对应位置更新。

如果还没有能力清单，使用以下模板创建：

- `docs/examples/product-capability-template.md`

目标不是创建另一套规划栈，而是让隐藏的能力约束持久化且可复用。

## 不可违反规则

- 不要虚构产品事实。明确标记未解决的问题。
- 区分用户可见的承诺和实现细节。
- 明确区分固定政策、架构偏好和仍未确定的内容。
- 如果需求与现有代码仓库约束冲突，明确说明而非隐瞒。
- 优先使用一个可复用的能力产物，而非零散的临时笔记。

## 输入

仅读取所需内容：

1. 产品意图
   - issue、讨论、PRD、路线图笔记、创始人消息
2. 当前架构
   - 相关仓库文档、契约、schema、路由、现有工作流
3. 现有能力上下文
   - `PRODUCT.md`、设计文档、RFC、迁移笔记、运营模型文档
4. 交付约束
   - 权限、计费、合规、灰度发布、向后兼容性、性能、评审政策

## 核心工作流

### 1. 重述能力

将需求压缩为一个精确的陈述：

- 用户或运营者是谁
- 上线后新增了什么能力
- 带来了什么结果变化

如果这个陈述不够明确，实现过程就会出现偏差。

### 2. 梳理能力约束

提取实施前必须满足的约束：

- 业务规则
- 范围边界
- 不变量
- 信任边界
- 数据所有权
- 生命周期流转
- 灰度发布/迁移要求
- 故障和恢复预期

这些内容通常只存在于高级工程师的记忆中。

### 3. 定义面向实现的契约

生成SRS风格的能力规划，包含：

- 能力概述
- 明确的非目标
- 参与者和暴露面
- 所需状态和流转
- 接口/输入/输出
- 数据模型影响
- 安全/计费/政策约束
- 可观测性和运营要求
- 阻碍实施的未解决问题

### 4. 转化为执行方案

以明确的交接内容结束：

- 可直接开始实施
- 需要先进行架构评审
- 需要先获得产品澄清

如果有用，指向后续的ECC原生流程：

- `project-flow-ops`
- `workspace-surface-audit`
- `api-connector-builder`
- `dashboard-builder`
- `tdd-workflow`
- `verification-loop`

## 输出格式

按以下顺序返回结果：

```text
CAPABILITY
- 一段式重述

CONSTRAINTS
- 固定规则、不变量和边界

IMPLEMENTATION CONTRACT
- 参与者
- 暴露面
- 状态和流转
- 接口/数据影响

NON-GOALS
- 本流程明确不负责的内容

OPEN QUESTIONS
- 仍需解决的阻塞问题或产品决策

HANDOFF
- 下一步应该做什么，以及应该使用哪个ECC流程
```

## 预期成果

- 产品意图现在足够具体，可以直接实施，不会在PR开发过程中才发现隐藏约束。
- 工程评审有了持久化的产物，不再依赖记忆或Slack上下文。
- 生成的规划可在Claude Code、Codex、Cursor、OpenCode和ECC 2.0规划工具中复用。

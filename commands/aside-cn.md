# Aside 命令

在任务执行过程中提问并获得即时、聚焦的答案，然后从你离开的地方继续。当前任务、文件和上下文永远不会被修改。

## 何时使用

- 当Claude正在工作时你对某事感到好奇，且不想失去当前节奏
- 你需要快速解释Claude当前正在编辑的代码
- 你希望在不偏离任务的情况下获得对某个决策的第二意见或澄清
- 你需要在Claude继续之前理解某个错误、概念或模式
- 你想问与当前任务无关的内容，而无需启动新会话

## 使用方法

```
/aside <your question>
/aside what does this function actually return?
/aside is this pattern thread-safe?
/aside why are we using X instead of Y here?
/aside what's the difference between foo() and bar()?
/aside should we be worried about the N+1 query we just added?
```

## 流程

### 步骤1：冻结当前任务状态

在回答任何内容之前，请在脑海中记下：
- 活动任务是什么？（正在处理什么文件、功能或问题）
- 调用`/aside`时正在执行什么步骤？
- 接下来要做什么？

在aside期间，请勿触碰、编辑、创建或删除任何文件。

### 步骤2：直接回答问题

以最简洁的形式回答问题，同时保持完整和有用。

- 先给出答案，再说明原因
- 保持简短——如果需要完整解释，可在任务完成后提供更深入的说明
- 如果问题与当前正在处理的文件或代码相关，请精确引用（相关时包含文件路径和行号）
- 如果回答需要读取文件，请读取——但仅读取，永远不要写入

响应格式如下：

```
ASIDE: [简要重述问题]

[你的答案]

— 返回任务：[正在执行的工作的单行描述]
```

### 步骤3：恢复主任务

给出答案后，立即从暂停的位置继续活动任务。除非aside答案显示存在阻碍或有理由重新考虑当前方法（请参阅边缘情况），否则无需请求恢复许可。

## 边缘情况

**未提供问题（`/aside`后面没有内容）：**
响应：
```
ASIDE: 未提供问题

你想了解什么？（提出你的问题，我会在不丢失当前任务上下文的情况下回答）

— 返回任务：[正在执行的工作的单行描述]
```

**问题显示当前任务存在潜在问题：**
在恢复之前明确标记：
```
ASIDE: [答案]

WARNING: 注意：此答案表明当前方法存在[问题]。你想在继续之前解决此问题，还是按计划进行？
```
等待用户决定后再恢复。

**问题实际上是任务重定向（不是侧边问题）：**
如果问题暗示更改正在构建的内容（例如：`/aside actually, let's use Redis instead`），请说明：
```
ASIDE: 这听起来像是方向变更，而不仅仅是侧边问题。
你希望：
  (a) 仅作为信息回答，保留当前计划
  (b) 暂停当前任务并更改方法
```
等待用户回答——不要做假设。

**问题与当前打开的文件或代码相关：**
根据实时上下文回答。如果文件在会话早期已被读取，请直接引用。如果没有，请现在读取（只读）并使用文件:行号引用回答。

**无活动任务（调用`/aside`时没有正在进行的工作）：**
仍使用标准包装，以保持响应格式一致：
```
ASIDE: [简要重述问题]

[你的答案]

— 返回任务：无活动任务可恢复
```

**问题需要长答案：**
简洁给出核心答案，然后提议：
```
以上是简短版本。在我们完成[当前任务]后需要更深入的解释吗？
```

**连续多个`/aside`问题：**
按顺序回答每个问题。最后一个答案后，恢复主任务。在一系列aside过程中不要丢失任务状态。

**Aside答案暗示需要更改代码：**
记录需要的更改，但不要在aside期间进行：
```
ASIDE: [答案]

值得修复：[应更改的内容]。除非你现在想解决，否则我会在当前任务后标记此问题。
```

**问题不明确或太模糊：**
提出一个澄清问题——获取回答所需信息的最短问题。不要问多个问题。

## 示例输出

```
User: /aside what does fetchWithRetry() actually do?

ASIDE: fetchWithRetry() 做什么？

fetchWithRetry()（src/api/retry.ts:12）最多尝试3次请求，使用指数退避（250ms → 500ms → 1s）。它仅在5xx响应和网络错误时重试——4xx错误被视为最终结果，不会重试。

— 返回任务：重构 src/middleware/auth.ts 中的auth中间件
```

```
User: /aside is the approach we're taking thread-safe?

ASIDE: 当前方法是线程安全的吗？

否——src/cache/store.ts:34 中的共享缓存对象在没有锁的情况下被修改。在并发请求下这是竞态条件。在单进程Node.js服务器中风险较低，但在使用worker线程或集群时会是一个实际问题。

WARNING: 注意：这可能会影响我们正在构建的功能。你想现在解决此问题，还是继续并在后续修复？
```

## 注意事项

- 永远不要在aside期间修改文件——仅允许只读访问
- Aside是会话暂停，不是新任务——必须始终恢复原始任务
- 保持答案聚焦：目标是快速解除用户的阻塞，而不是讲授课程
- 如果aside引发更大的讨论，先完成当前任务，除非aside显示存在阻碍
- 除非与任务结果明确相关，否则Asides不会保存到会话文件中

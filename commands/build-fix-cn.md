# 构建与修复

通过最小、安全的变更逐步修复 build 和 type 错误。

## 步骤 1：检测构建系统

识别项目的构建工具并运行构建：

| 标识符 | Build Command |
|-----------|---------------|
| `package.json` 包含 `build` 脚本 | `npm run build` 或 `pnpm build` |
| `tsconfig.json` (仅 TypeScript) | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall -q .` 或 `mypy .` |

## 步骤 2：解析并分组错误

1. 运行 build 命令并捕获 stderr 输出
2. 按文件路径分组错误
3. 按依赖顺序排序（先修复 imports/types 再修复逻辑错误）
4. 统计总错误数用于进度跟踪

## 步骤 3：修复循环（一次修复一个错误）

针对每个错误：

1. **读取文件** — 使用 Read 工具查看错误上下文（错误前后10行）
2. **诊断** — 识别根本原因（缺失导入、错误类型、语法错误）
3. **最小修复** — 使用 Edit 工具进行最小的变更解决错误
4. **重新运行构建** — 验证错误已消除且未引入新错误
5. **进入下一个** — 继续处理剩余错误

## 步骤 4：防护规则

出现以下情况时停止并询问用户：
- 一个修复引入的错误比解决的多
- 同一个错误在3次尝试后仍然存在（可能是更深层的问题）
- 修复需要架构变更（不仅仅是构建修复）
- 构建错误源于缺失依赖（需要 `npm install`、`cargo add` 等）

## 步骤 5：总结

展示结果：
- 已修复的错误（包含文件路径）
- 剩余的错误（如果有）
- 引入的新错误（应该为0）
- 对未解决问题的后续步骤建议

## 恢复策略

| 场景 | 操作 |
|-----------|--------|
| 缺失 module/import | 检查包是否已安装；建议安装命令 |
| Type 不匹配 | 读取两个类型定义；修复更窄的类型 |
| 循环依赖 | 使用导入图识别循环；建议提取公共部分 |
| 版本冲突 | 检查 `package.json` / `Cargo.toml` 中的版本约束 |
| Build 工具配置错误 | 读取配置文件；与可用的默认配置对比 |

一次修复一个错误保证安全。优先最小 diff 而非重构。

# /learn-eval - 提取、评估、然后保存

扩展了 `/learn` 功能，增加了质量关口、保存位置决策以及知识放置感知能力，在写入任何 skill 文件前执行这些检查。

## 提取内容

寻找以下内容：

1. **Error Resolution Patterns** — 根本原因 + 修复方案 + 可复用性
2. **Debugging Techniques** — 非直观步骤、工具组合使用方式
3. **Workarounds** — 库的特殊行为、API 限制、特定版本的修复方案
4. **Project-Specific Patterns** — 约定、架构决策、集成模式

## 流程

1. 审阅会话内容，寻找可提取的模式
2. 识别最有价值/可复用的洞察

3. **确定保存位置：**
   - 提问："这个模式在其他项目中是否有用？"
   - **Global** (`~/.claude/skills/learned/`): 可在2个以上项目中使用的通用模式（bash 兼容性、LLM API 行为、调试技巧等）
   - **Project** (当前项目中的 `.claude/skills/learned/`): 项目特定知识（特定配置文件的特殊行为、项目专属架构决策等）
   - 不确定时选择 Global（将 Global 移动到 Project 比反向操作更容易）

4. 使用以下格式编写 skill 文件草稿：

```markdown
---
name: pattern-name
description: "不超过130个字符"
user-invocable: false
origin: auto-extracted
---

# [描述性模式名称]

**提取日期:** [日期]
**适用场景:** [简要描述该模式的适用情况]

## 问题
[该模式解决的具体问题 - 请明确描述]

## 解决方案
[模式/技术/绕过方案 - 包含代码示例]

## 使用时机
[触发条件]
```

5. **质量关口 — 检查清单 + 整体评估**

   ### 5a. 必备检查清单（通过实际读取文件进行验证）

   评估草稿前必须执行以下**所有**检查：

   - [ ] 通过关键词 Grep `~/.claude/skills/` 和相关项目的 `.claude/skills/` 文件，检查内容是否重复
   - [ ] 检查 MEMORY.md（项目和全局）是否有重复内容
   - [ ] 考虑是否可以追加到现有 skill 中
   - [ ] 确认这是可复用模式，而非一次性修复

   ### 5b. 整体评估

   综合检查清单结果和草稿质量，选择以下**一个**选项：

   | Verdict | 含义 | 下一步操作 |
   |---------|------|------------|
   | **Save** | 独特、具体、范围清晰 | 继续执行第6步 |
   | **Improve then Save** | 有价值但需要优化 | 列出改进点 → 修改 → 重新评估（仅一次） |
   | **Absorb into [X]** | 应追加到现有 skill 中 | 展示目标 skill 和新增内容 → 第6步 |
   | **Drop** | 过于琐碎、重复或太抽象 | 解释原因并终止流程 |

   **评估参考维度**（供评估参考，不打分）：

   - **Specificity & Actionability**: 包含可直接使用的代码示例或命令
   - **Scope Fit**: 名称、触发条件和内容保持一致，聚焦于单个模式
   - **Uniqueness**: 提供现有 skill 未覆盖的价值（基于检查清单结果）
   - **Reusability**: 在未来会话中存在实际触发场景

6. **基于评估结果的确认流程**

   - **Improve then Save**: 展示需要的改进点 + 修改后的草稿 + 重新评估后的检查清单/结果；如果重新评估结果为 **Save**，经用户确认后保存，否则按照新结果执行
   - **Save**: 展示保存路径 + 检查清单结果 + 1行评估理由 + 完整草稿 → 经用户确认后保存
   - **Absorb into [X]**: 展示目标路径 + 新增内容（diff 格式） + 检查清单结果 + 评估理由 → 经用户确认后追加
   - **Drop**: 仅展示检查清单结果 + 理由（无需确认）

7. 保存/追加到确定的位置

## 第5步输出格式

```
### 检查清单
- [x] skills/ grep：无重复（或：发现重复 → 详细信息）
- [x] MEMORY.md：无重复（或：发现重复 → 详细信息）
- [x] 现有 skill 追加：适合新建文件（或：应追加到 [X]）
- [x] 可复用性：已确认（或：一次性问题 → Drop）

### 评估结果：Save / Improve then Save / Absorb into [X] / Drop

**理由：**（1-2句话解释评估结果）
```

## 设计原理

该版本将之前的5维度数值评分体系（Specificity、Actionability、Scope Fit、Non-redundancy、Coverage 按1-5分打分）替换为基于检查清单的整体评估系统。现代前沿模型（Opus 4.6+）具备强大的上下文判断能力——将丰富的定性信号强制转换为数值分数会丢失细节，可能产生误导性的总分。整体评估方法让模型可以自然权衡所有因素，做出更准确的保存/丢弃决策，同时明确的检查清单确保不会遗漏任何关键检查项。

## 注意事项

- 不要提取琐碎的修复内容（拼写错误、简单语法错误）
- 不要提取一次性问题（特定 API 中断等）
- 聚焦于能在未来会话中节省时间的模式
- 保持 skill 内容聚焦 —— 每个 skill 对应一个模式
- 当评估结果为 Absorb 时，追加到现有 skill 中，而非创建新文件

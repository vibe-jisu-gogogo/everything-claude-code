# PRP Plan

创建详细、自包含的实现计划，捕获所有代码库模式、约定和上下文，支持一次性完成功能实现。

**核心理念**：一个优秀的计划应包含实现所需的所有信息，无需进一步提问。每个模式、每个约定、每个注意事项——一次性捕获，全程引用。

**黄金法则**：如果你在实现过程中需要搜索代码库，请现在就将这些知识记录到计划中。

---

## 阶段 0 — DETECT

从 `$ARGUMENTS` 确定输入类型：

| 输入模式 | 检测结果 | 操作 |
|---|---|---|
| 路径以 `.prd.md` 结尾 | PRD 文件路径 | 解析 PRD，查找下一个待处理阶段 |
| 包含 "Implementation Phases" 的 `.md` 文件路径 | 类 PRD 文档 | 解析阶段，查找下一个待处理项 |
| 其他任意文件路径 | 参考文件 | 读取文件获取上下文，视为自由格式输入 |
| 自由格式文本 | 功能描述 | 直接进入阶段 1 |
| 空/空白 | 无输入 | 询问用户需要规划什么功能 |

### PRD 解析（当输入为 PRD 时）

1. 使用 `cat "$PRD_PATH"` 读取 PRD 文件
2. 解析 **Implementation Phases** 部分
3. 按状态查找阶段：
   - 查找 `pending` 状态的阶段
   - 检查依赖链（某个阶段可能依赖于之前的阶段完成）
   - 选择 **下一个符合条件的待处理阶段**
4. 从选中的阶段提取：
   - 阶段名称和描述
   - 验收标准
   - 对先前阶段的依赖
   - 任何范围说明或约束
5. 使用阶段描述作为要规划的功能

如果没有剩余的待处理阶段，报告所有阶段已完成。

## 阶段 1 — PARSE

提取并明确功能需求。

### 功能理解

从输入（PRD 阶段或自由格式描述）中识别：

- **What** 正在构建什么（具体交付物）
- **Why** 为什么重要（用户价值）
- **Who** 谁使用它（目标用户/系统）
- **Where** 它适合在哪里（代码库的哪个部分）

### User Story

格式如下：
```
As a [用户类型],
I want [功能],
So that [收益].
```

### 复杂度评估

| 级别 | 指标 | 典型范围 |
|---|---|---|
| **Small** | 单个文件，隔离变更，无新依赖 | 1-3 个文件，<100 行 |
| **Medium** | 多个文件，遵循现有模式，少量新概念 | 3-10 个文件，100-500 行 |
| **Large** | 跨领域关注点，新模式，外部集成 | 10+ 个文件，500+ 行 |
| **XL** | 架构变更，新子系统，需要迁移 | 20+ 个文件，考虑拆分 |

### 歧义检查点

如果以下任何一项不明确，**停止并询问用户** 后再继续：

- 核心交付物模糊
- 成功标准未定义
- 存在多种有效解释
- 技术方案有重大未知项

不要猜测。直接询问。基于假设构建的计划在实现过程中会失败。

## 阶段 2 — EXPLORE

收集深入的代码库信息。针对以下每个类别直接搜索代码库。

### 代码库搜索（8 个类别）

对于每个类别，使用 grep、find 和文件读取进行搜索：

1. **Similar Implementations** — 查找与计划功能类似的现有功能。寻找类似的模式、端点、组件或模块。
2. **Naming Conventions** — 确定代码库相关区域中文件、函数、变量、类和导出的命名方式。
3. **Error Handling** — 查找类似代码路径中如何捕获、传播、记录错误并返回给用户。
4. **Logging Patterns** — 确定记录什么内容、使用什么级别以及什么格式。
5. **Type Definitions** — 查找相关类型、接口、模式及其组织方式。
6. **Test Patterns** — 查找类似功能的测试方式。注意测试文件位置、命名、设置/拆卸模式和断言风格。
7. **Configuration** — 查找相关配置文件、环境变量和功能开关。
8. **Dependencies** — 识别类似功能使用的包、导入和内部模块。

### 代码库分析（5 个追踪点）

读取相关文件以追踪：

1. **Entry Points** — 请求/操作如何进入系统并到达你要修改的区域？
2. **Data Flow** — 数据如何在相关代码路径中流动？
3. **State Changes** — 哪些状态被修改，在哪里修改？
4. **Contracts** — 必须遵守哪些接口、API 或协议？
5. **Patterns** — 使用了哪些架构模式（repository、service、controller 等）？

### 统一发现表

将发现结果编译为单一参考：

| 类别 | 文件:行号 | 模式 | 关键代码片段 |
|---|---|---|---|
| 命名 | `src/services/userService.ts:1-5` | 服务使用 camelCase，类型使用 PascalCase | `export class UserService` |
| 错误 | `src/middleware/errorHandler.ts:10-25` | 自定义 AppError 类 | `throw new AppError(...)` |
| ... | ... | ... | ... |

## 阶段 3 — RESEARCH

如果功能涉及外部库、API 或不熟悉的技术：

1. 搜索官方文档
2. 查找使用示例和最佳实践
3. 识别版本特定的注意事项

每个发现的格式如下：

```
KEY_INSIGHT: [你学到的内容]
APPLIES_TO: [这会影响计划的哪个部分]
GOTCHA: [任何警告或版本特定问题]
```

如果功能仅使用已充分理解的内部模式，跳过此阶段并注明："无需外部研究——功能使用已建立的内部模式。"

## 阶段 5 — ARCHITECT

### 战略设计

定义实现方案：

- **Approach**: 高层策略（例如："遵循现有 repository 模式添加新的 service 层"）
- **Alternatives Considered**: 评估了哪些其他方案，以及为什么拒绝它们
- **Scope**: 将要构建内容的具体边界
- **NOT Building**: 明确列出超出范围的内容（防止实现过程中范围蔓延）

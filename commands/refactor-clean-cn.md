# 重构清理

每一步都通过测试验证，安全地识别并移除死代码。

## 步骤 1：检测死代码

根据项目类型运行分析工具：

| Tool | What It Finds | Command |
|------|--------------|---------|
| knip | 未使用的 exports、文件、依赖 | `npx knip` |
| depcheck | 未使用的 npm 依赖 | `npx depcheck` |
| ts-prune | 未使用的 TypeScript exports | `npx ts-prune` |
| vulture | 未使用的 Python 代码 | `vulture src/` |
| deadcode | 未使用的 Go 代码 | `deadcode ./...` |
| cargo-udeps | 未使用的 Rust 依赖 | `cargo +nightly udeps` |

如果没有可用的工具，使用 Grep 查找零导入的 exports：
```
# Find exports, then check if they're imported anywhere
```

## 步骤 2：分类结果

将结果按安全等级分类：

| Tier | Examples | Action |
|------|----------|--------|
| **SAFE** | 未使用的工具函数、测试辅助函数、内部函数 | 可放心删除 |
| **CAUTION** | 组件、API 路由、中间件 | 验证没有动态导入或外部消费者 |
| **DANGER** | 配置文件、入口文件、类型定义 | 修改前先仔细调查 |

## 步骤 3：安全删除循环

对于每个 SAFE 项：

1. **运行完整测试套件** — 建立基准线（全部通过）
2. **删除死代码** — 使用 Edit 工具进行精确移除
3. **重新运行测试套件** — 验证没有破坏任何功能
4. **如果测试失败** — 立即用 `git checkout -- <file>` 回滚并跳过此项
5. **如果测试通过** — 继续处理下一项

## 步骤 4：处理 CAUTION 项

删除 CAUTION 项之前：
- 搜索动态导入：`import()`, `require()`, `__import__`
- 搜索字符串引用：配置中的路由名称、组件名称
- 检查是否从公共包 API 导出
- 验证没有外部消费者（如果已发布，检查依赖项）

## 步骤 5：合并重复项

移除死代码后，查找：
- 近似重复的函数（相似度 >80%）— 合并为一个
- 冗余的类型定义 — 合并
- 没有价值的包装函数 — 内联它们
- 没有作用的重新导出 — 移除间接层

## 步骤 6：总结

报告结果：

```
死代码清理
──────────────────────────────
已删除:   12 个未使用函数
           3 个未使用文件
           5 个未使用依赖
已跳过:   2 项（测试失败）
节省:     ~450 行代码被移除
──────────────────────────────
所有测试通过 PASS:
```

## 规则

- **永远不要在未运行测试的情况下删除**
- **一次只删除一项** — 原子变更让回滚更容易
- **不确定就跳过** — 保留死代码总比破坏生产环境好
- **清理时不要同时重构** — 关注点分离（先清理，后重构）

# 保存会话命令

捕获本次会话中发生的所有内容——已构建的内容、运行正常的部分、失败的部分、剩余待完成的工作——并将其写入带日期的文件中，以便下次会话可以准确地从上次中断的地方继续。

## 使用场景

- 工作会话结束时，关闭 Claude Code 之前
- 即将达到上下文限制前（先运行此命令，然后开启新会话）
- 解决了一个你希望记住的复杂问题之后
- 任何你需要将上下文传递给未来会话的场景

## 执行流程

### 步骤 1：收集上下文

在写入文件前，收集以下内容：

- 读取本次会话中修改过的所有文件（使用 git diff 或从对话中回忆）
- 回顾讨论过的内容、尝试过的方案和做出的决策
- 记录遇到的所有错误以及它们是如何被解决的（或未解决）
- 如果相关，检查当前 test/build 的状态

### 步骤 2：如果会话文件夹不存在则创建它

在用户的 Claude 主目录下创建标准会话文件夹：

```bash
mkdir -p ~/.claude/session-data
```

### 步骤 3：写入会话文件

创建 `~/.claude/session-data/YYYY-MM-DD-<short-id>-session.tmp`，使用实际当前日期和符合 `session-manager.js` 中 `SESSION_FILENAME_REGEX` 规则的短 id：

- 兼容字符：字母 `a-z` / `A-Z`、数字 `0-9`、连字符 `-`、下划线 `_`
- 兼容最小长度：1个字符
- 新文件推荐样式：小写字母、数字和连字符，长度8+字符以避免冲突

有效示例：`abc123de`、`a1b2c3d4`、`frontend-worktree-1`、`ChezMoi_2`
新文件避免使用：`A`、`test_id1`、`ABC123de`

完整有效文件名示例：`2024-01-15-abc123de-session.tmp`

旧版文件名 `YYYY-MM-DD-session.tmp` 仍然有效，但新会话文件应优先使用短 id 形式以避免同一天的命名冲突。

### 步骤 4：使用以下所有部分填充文件

如实填写每个部分。不要跳过任何部分——如果某个部分确实没有内容，请写"暂无内容"或"N/A"。不完整的文件比诚实的空部分更糟糕。

### 步骤 5：向用户展示文件

写入完成后，展示完整内容并询问：

```
会话已保存到 [会话文件的实际解析路径]

看起来是否准确？在结束前有什么需要更正或补充的吗？
```

等待确认。如果有要求请进行编辑。

## 我们正在构建的内容

用户身份验证系统用于 Next.js 应用。用户通过邮箱/密码注册，获得存储在 httpOnly cookie（而非 localStorage）中的 JWT，受保护的路由通过中间件检查有效 token。目标是在浏览器刷新时保持会话持久化，同时不将 token 暴露给 JavaScript。

---

## 运行正常的部分（附证据）

- **`/api/auth/register` 端点** — 确认依据：Postman POST 请求返回 200 及用户对象，Supabase 面板中可见行记录，bcrypt 哈希存储正确
- **`lib/auth.ts` 中的 JWT 生成** — 确认依据：单元测试通过 (`npm test -- auth.test.ts`)，在 jwt.io 解码后的 token 显示正确的 payload
- **密码哈希** — 确认依据：测试中 `bcrypt.compare()` 返回 true

---

## 运行失败的部分（及原因）

- **Next-Auth 库** — 失败原因：与我们自定义的 Prisma adapter 冲突，每次请求都抛出"Cannot use adapter with credentials provider in this configuration"错误。不值得调试——对我们的设置来说过于 opinionated。
- **将 JWT 存储在 localStorage 中** — 失败原因：SSR 渲染发生在 localStorage 可用之前，导致每次页面加载都出现 React hydration mismatch 错误。此方案与 Next.js SSR 根本不兼容。

---

## 尚未尝试的内容

- 使用 `next/headers` 中的 `cookies()` 在服务端组件中读取 token
- 编写 middleware.ts 通过检查 cookie 存在性来保护路由

- **自定义身份验证而非使用 Next-Auth** — 原因：Next-Auth 与我们的 Prisma 设置冲突，不值得花费精力调试

## 注意事项

- 每个会话都有自己的文件——永远不要追加到之前会话的文件中
- "运行失败的部分"是最关键的——如果没有它，未来会话会盲目重试失败的方案
- 如果用户要求在会话中途保存（不仅是在结束时），保存目前已知的内容并清晰标记进行中的条目
- 该文件旨在供 Claude 在下次会话开始时通过 `/resume-session` 读取
- 使用标准全局会话存储：`~/.claude/session-data/`
- 所有新会话文件优先使用短 id 文件名形式（`YYYY-MM-DD-<short-id>-session.tmp`）

## 发现

### 中等

1. `skills/continuous-learning-v2/hooks/observe.sh`

新的 `CLAUDE_CODE_ENTRYPOINT` 防护使用了一个有限的已知非 `cli` 值白名单（`sdk-ts`、`sdk-py`、`sdk-cli`、`mcp`、`remote`）。

这留下了一个前向兼容性漏洞：任何未来的非 `cli` entrypoint 值都会被漏掉并被视为交互式。这会重新引入该 PR 试图防止的那类自动化会话观察问题。

更安全的规则是：

- 只允许 `cli`
- 将所有其他显式 entrypoint 视为自动化
- 当变量未设置时，保持默认回退为 `cli`

建议的形式：

```bash
case "${CLAUDE_CODE_ENTRYPOINT:-cli}" in
  cli) ;;
  *) exit 0 ;;
esac
```

## 合并建议

`合并前需要一项后续更改。`

该 PR 的方向是正确的：

- 它关闭了 `observer-loop.sh` 中的 ECC 自我观察循环
- 它在 `observe.sh` 的正确区域添加了多层防护
- 它已经解决了成本优先排序和 skip-path 修剪问题

但在合并前应该泛化 entrypoint 防护，这样当 Claude Code 引入额外的非交互式 entrypoints 时，自动化过滤器不会静默失效。

## 残留风险

- 新的 shell 防护行为周围仍然没有专门的回归测试覆盖，因此最终合并应至少包含针对 entrypoint 和 skip-path 情况的一次可执行验证。

# Capability Surface Selection

本文档作为功能归属的路由指南，用于决定一个功能应该放在 rule、skill、MCP server，还是普通的 CLI/API 工作流中。

ECC 并不将这些实现层面视为可互换的。目标是将每个功能放在最窄的实现层面上，以保持正确性、控制 token 成本，并且不会产生不必要的运行时或供应链负担。

## 简版说明

- `rules/` 用于确定性的、始终生效的约束，当路径或事件匹配时应被注入。
- `skills/` 用于按需工作流、更丰富的操作手册，以及仅在相关时才应加载的 token 密集型指导。
- `MCP` 用于交互式结构化功能，这些功能能从跨会话或客户端的长期工具/资源层面中受益。
- 本地 `CLI` 或仓库脚本用于不需要持久化服务器的简单确定性操作。
- 在 skill 中直接进行 `API` 调用用于狭义的远程操作，此时完整的 MCP server 会比问题本身更沉重。

## 决策顺序

按顺序询问以下问题：

1. 是否应该在每次路径或事件匹配时都发生，且不涉及模型判断？
   - 使用 `rule`。
2. 是否主要是一个操作手册、工作流或建议层，仅在任务实际需要时才应加载？
   - 使用 `skill`。
3. 该功能是否需要一个结构化的交互式工具/资源接口，供多个 harness 或客户端重复调用？
   - 使用 `MCP`。
4. 是否是一个可以作为脚本运行而无需保持服务器存活的简单本地操作？
   - 使用本地 `CLI` 入口或仓库脚本，必要时用 skill 包装。
5. 是否只是更大工作流中的一个狭义远程集成步骤？
   - 从 skill 或脚本中直接调用外部 `API`。

## 各实现层面指南

### Rules

将 rules 用于：

- 路径范围的代码不变量
- 安全底线和权限约束
- 应始终适用的 harness/运行时约束
- 不应依赖模型裁量权的确定性提醒

不要将 rules 用于：

- 会使每个匹配的编辑变得臃肿的大型操作手册
- 可选工作流
- 仅在某些时候才有意义的昂贵领域上下文

### Skills

将 skills 用于：

- 多步骤工作流
- 需要大量判断的指导
- 昂贵到需要按需加载的领域操作手册
- 跨脚本、API、MCP 工具和相邻 skills 的编排

不要将 skills 作为真正需要确定性路由的静态不变量的倾倒场。

### MCP

在以下情况下使用 MCP：

- 结构化的工具输入/输出
- 可重用的资源或提示
- 跨客户端的重复使用
- 应在 Claude Code、Codex、Cursor、OpenCode 和相关 harness 中工作的稳定接口
- 长期运行的服务器进程值得运营开销

在以下情况下避免使用 MCP：

- 任务是一次性本地命令
- 服务器唯一要做的事情就是执行一次 shell 命令
- 服务器带来的安装/运行时负担超过了产品价值

### CLI / 仓库脚本

在以下情况下优先使用本地脚本或 CLI：

- 操作是确定性的
- 启动成本低
- 工作流主要是本地的
- 暴露持久化的工具/资源层面没有好处

这通常是以下情况的正确选择：

- lint/test/build 包装器
- 本地转换
- 小型安装程序
- 每次调用运行一次的内容生成

### 直接 API 调用

在以下情况下优先在现有 skill 或脚本中进行直接 API 调用：

- 集成范围很窄
- 远程操作是更大工作流的一部分
- 还不需要可重用的传输层面

如果相同的远程集成变得核心、重复且多客户端，这就是将其升级为 MCP 层面的信号。

## 成本和可靠性偏好

当两个选项都可行时：

- 优先选择更小的运行时层面
- 优先选择更低的 token 开销
- 优先选择外部活动部件更少的路径
- 优先选择 ECC 原生包装而非引入另一个第三方依赖

不要将外部插件或包依赖标准化为一流的 ECC 层面，除非该功能显然值得维护、安全和安装负担。

## 仓库策略

从外部仓库引入想法时：

- 复制底层想法，而不是外部依赖
- 将其重新打包为 ECC 原生的 rule、skill、script 或 MCP 层面
- 如果功能已为 ECC 进行了实质性扩展或重塑，请重命名
- 避免发布要求用户安装无关第三方包的说明，除非该依赖是有意的、经过审计的，并且是工作流的核心。

## 示例

- 应始终适用于 `api/**` 编辑的后端认证不变量：
  - `rule`
- 更深入的 API 设计和分页操作手册：
  - `skill`
- 跨多个 harness 使用的可重用远程搜索层面：
  - `MCP`
- 读取本地文件并写入报告的一次性仓库分析器：
  - 本地 `CLI` 或脚本，可选由 `skill` 包装
- 更广泛的客户运营工作流中的单个计费门户会话创建步骤：
  - 工作流内的直接 `API` 调用

## 实用启发式

如果不确定，从小处开始：

- 对于确定性不变量，从 `rule` 开始
- 对于指导/工作流，从 `skill` 开始
- 对于一次性执行，从脚本开始
- 仅当结构化服务器边界明显物有所值时，才升级到 `MCP`

# 编码风格

## 不变性（重要）

始终创建新对象，不修改现有对象：

```
// 伪代码
错误:  modify(original, field, value) → 原地修改 original
正确: update(original, field, value) → 返回应用了变更的新副本
```

理由: 不可变数据防止隐藏的副作用，使调试更容易，并支持安全的并发处理。

## 文件组织

许多小文件 > 少数大文件:
- 高内聚，低耦合
- 通常 200-400 行，最大 800 行
- 从大模块中提取工具函数
- 按功能/领域组织，而不是按类型

## 错误处理

始终全面处理错误:
- 在所有层级显式处理错误
- 在面向 UI 的代码中提供用户友好的错误消息
- 在服务器端记录详细的错误上下文
- 不要静默忽略错误

## 输入验证

始终在系统边界进行验证:
- 在处理前验证所有用户输入
- 尽可能使用基于 schema 的验证
- 用清晰的错误消息快速失败
- 永远不要信任外部数据（API 响应、用户输入、文件内容）

## 代码质量检查清单

在将工作标记为完成之前:
- [ ] 代码易读且命名恰当
- [ ] 函数很小（少于 50 行）
- [ ] 文件有明确的焦点（少于 800 行）
- [ ] 没有深层嵌套（4 层或更少）
- [ ] 适当的错误处理
- [ ] 没有硬编码值（使用常量或配置）
- [ ] 没有变更（使用不可变模式）

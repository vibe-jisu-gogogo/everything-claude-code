# 编码风格

## 不变性（重要）

始终创建新对象，永远不要修改现有对象：

```
// 伪代码
错误示例:  modify(original, field, value) → 直接修改原始对象
正确示例: update(original, field, value) → 返回应用了更改的新副本
```

理由：不变数据可防止隐藏的副作用，简化调试，并支持安全的并发。

## 文件组织

许多小文件 > 少量大文件：
- 高内聚，低耦合
- 通常 200-400 行，最多 800 行
- 从大模块中分离工具函数
- 按功能/领域而非类型组织

## 错误处理

始终全面处理错误：
- 在所有级别显式处理错误
- 在 UI 代码中提供用户友好的错误消息
- 在服务器端记录详细的错误上下文
- 永远不要静默忽略错误

## 输入验证

始终在系统边界进行验证：
- 处理前验证所有用户输入
- 尽可能使用基于 schema 的验证
- 使用清晰的错误消息快速失败
- 永远不要信任外部数据（API 响应、用户输入、文件内容）

## 代码质量检查清单

完成工作前检查：
- [ ] 代码是否易读且命名恰当
- [ ] 函数是否较小 (<50 行)
- [ ] 文件是否聚焦 (<800 行)
- [ ] 是否没有深度嵌套 (>4 层)
- [ ] 是否有适当的错误处理
- [ ] 是否没有硬编码值（使用常量或配置）
- [ ] 是否没有变异（使用不可变模式）

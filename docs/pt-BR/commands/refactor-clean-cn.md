# 重构清理

安全地识别和移除死代码，每一步都进行测试验证。

## 步骤 1：检测死代码

根据项目类型运行分析工具：

| Tool | What It Finds | Command |
|------|--------------|---------|
| knip | 未使用的导出、文件、依赖 | `npx knip` |
| depcheck | 未使用的 npm 依赖 | `npx depcheck` |
| ts-prune | 未使用的 TypeScript 导出 | `npx ts-prune` |
| vulture | 未使用的 Python 代码 | `vulture src/` |
| deadcode | 未使用的 Go 代码 | `deadcode ./...` |
| cargo-udeps | 未使用的 Rust 依赖 | `cargo +nightly udeps` |

如果没有可用的工具，使用 Grep 查找零引用的导出：
```
# 查找导出，然后检查是否在任何地方被导入
```

## 步骤 2：分类结果

将发现结果按安全级别分类：

| Tier | Examples | Action |
|------|----------|--------|
| **SAFE** | 未使用的工具函数、测试辅助函数、内部函数 | 放心删除 |
| **CAUTION** | 组件、API 路由、中间件 | 验证没有动态导入或外部使用者 |
| **DANGER** | 配置文件、入口点、类型定义 | 先调查再操作 |

## 步骤 3：安全移除循环

对于每个 SAFE 项目：

1. **运行完整的测试套件** — 建立基线（全部通过）
2. **删除死代码** — 使用 Edit 工具进行精确移除
3. **再次运行测试套件** — 验证没有破坏任何功能
4. **如果测试失败** — 立即使用 `git checkout -- <file>` 回滚并跳过此项目
5. **如果测试通过** — 继续下一个项目

## 步骤 4：处理 CAUTION 项目

删除 CAUTION 项目之前：
- 查找动态导入：`import()`、`require()`、`__import__`
- 查找字符串引用：路由名称、配置中的组件名称
- 验证是否通过包的公共 API 导出
- 检查是否没有外部使用者（依赖项，如果已发布）

## 步骤 5：合并重复项

移除死代码后，查找：
- 近乎重复的函数（>80% 相似）— 合并为一个
- 冗余的类型定义 — 合并
- 无价值的包装函数 — 内联
- 无目的的重新导出 — 移除间接层

## 步骤 6：总结

报告结果：

```
死代码清理
──────────────────────────────
已删除:   12 个未使用的函数
           3 个未使用的文件
           5 个未使用的依赖
已跳过:   2 个项目（测试失败）
节省:     约 450 行代码已移除
──────────────────────────────
所有测试通过 PASS:
```

## 规则

- **运行测试前绝对不要删除**
- **一次只删除一个** — 原子变更便于回滚
- **如有疑问，跳过** — 保留死代码总比破坏生产环境好
- **清理期间不要重构** — 分离职责（先清理，后重构）
